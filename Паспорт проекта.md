**Цель проекта:** создать удобный инструмент для регистрации, командообразования, обработки заявок и контроля состава команд.

**5\. Техническое задание**

**5.1. Общие сведения.** Система сопровождения хакатонов предназначена для автоматизации регистрации, формирования команд, обработки заявок и взаимодействия участников и организаторов в единой цифровой среде.  
**5.2. Основание для разработки.** В рамках проведения хакатонов ключевые процессы часто выполняются вручную или через разрозненные инструменты. Это приводит к высокой операционной нагрузке на организаторов, ошибкам, задержкам в коммуникации и отсутствию единой среды взаимодействия. Разработка единой системы на базе Telegram и Django позволит централизовать процессы, снизить нагрузку на организаторов и повысить удобство участия в хакатоне.

**5.3. Цели и задачи**

**Цели:**

• Создать единую цифровую среду сопровождения хакатона внутри Telegram.  
• Упростить регистрацию участников и поиск команды.  
• Обеспечить быстрый доступ к профилю участника и статуса его участия.  
• Предоставить организаторам удобный административный просмотр данных.

**Задачи:**

• Реализовать регистрацию участника и сохранение профиля.  
• Реализовать создание команды и подачу заявок в команды.  
• Реализовать Telegram Mini App с двумя экранами: профиль и поиск команды.  
• Реализовать административный просмотр участников, команд и технологических стеков.  
• Реализовать уведомления через Telegram-бота.  
• Обеспечить единый backend на Django для всех интерфейсов. • Реализовать выгрузку данных для организаторов (Excel / CSV).

**SMART-цель:**  
**S (Specific):** Разработать систему сопровождения хакатонов на базе Django, Telegram-бота и Mini App для автоматизации регистрации, формирования команд и управления заявками.  
**M (Measurable):** ≥ 90% участников проходят регистрацию через систему; снижение ручных операций организаторов ≥ 60%; время обработки пользовательского действия ≤ 5 секунд.  
**A (Achievable):** Реализуется с использованием выбранного технологического стека и MVP-подхода.  
**R (Relevant):** Соответствует задачам автоматизации и улучшения пользовательского опыта.  
**T (Time-bound):** Срок реализации: до 30.04.

**5.4. Состав системы и архитектура**

• Django backend — центральная серверная часть системы: хранение данных, бизнес-логика, проверка прав доступа, работа с профилями, командами и заявками.  
• Telegram-бот — запуск системы, первичная регистрация, уведомления, быстрые действия и переход в Mini App.  
• Telegram Mini App — только два пользовательских сценария: страница профиля и поиск команды.  
• Административный контур — Django admin для просмотра статусов пользователей, команд и стеков.

**5.6. Функциональные требования**

**Регистрация участника.**  
• Пользователь запускает бота командой /start.  
• Система предлагает пройти регистрацию и запрашивает ФИО, почту и технологический стек.  
• Данные сохраняются в базе данных Django.  
• После регистрации пользователь получает доступ к основным функциям.

**Профиль участника.**  
• Просмотр данных профиля.  
• Редактирование ФИО, почты и технологического стека.  
• Отображение текущего статуса: не в команде / участник команды / капитан.  
• Отображение привязанной команды, если пользователь состоит в команде.

**Создание команды.**  
• Участник без команды может создать команду.  
• При создании указываются название, описание, стек и вакансии.  
• После создания пользователь автоматически становится капитаном.  
• Команда появляется в списке открытых команд.

**Поиск команды.**  
• Пользователь просматривает список открытых команд.  
• Доступна фильтрация по технологическому стеку и вакансиям.  
• По каждой команде отображаются название, описание, стек, вакансии и статус открытого набора.  
• Пользователь может отправить заявку на вступление в команду.

**Управление заявками.**  
• Капитан видит заявки на вступление в свою команду.  
• По каждой заявке отображаются данные участника и его стек.  
• Капитан может принять или отклонить заявку.  
• После решения участник получает уведомление в Telegram.

**Уведомления.**  
• Уведомление капитану о новой заявке.  
• Уведомление участнику о принятии или отклонении заявки.  
• При необходимости — дополнительные уведомления по организационным событиям.

**5.7. Требования к Telegram Mini App**

**Назначение Mini App:** удобная работа с профилем и поиском команды без перегрузки бота сложными интерфейсами.  
**Состав экранов Mini App:**

• Страница профиля участника.  
• Страница поиска команды.

**Страница профиля.**  
• Отображение ФИО, почты, технологического стека и статуса участия.  
• Возможность редактирования профиля.  
• Отображение текущей командной принадлежности.

**Страница поиска команды.**  
• Список открытых команд.  
• Поиск и фильтрация по стеку и вакансиям.  
• Просмотр краткой информации по команде.  
• Подача заявки на вступление в команду.

**Авторизация Mini App:** выполняется через данные Telegram-пользователя с проверкой на стороне Django backend.

**5.8. Требования к административной части**

• Просмотр списка участников.  
• Просмотр статуса каждого участника: в команде / не в команде.  
• Просмотр технологического стека пользователя.  
• Просмотр состава команд и информации о капитане.  
• Отображение открытых и закрытых команд. • Запрос и получение выгрузки данных в формате Excel (.xlsx) или CSV.

**5.9. Требования к выгрузке данных (Excel / CSV)**

• Выгрузка доступна организатору и администратору.  
• Форматы выгрузки: Excel (.xlsx) и CSV.  
• Должны поддерживаться минимум два набора данных:

- Список участников (ФИО, контакты, технологический стек, статус в команде).  
- Список команд (название, капитан, количество участников, описание, вакансии, статус набора).  
  • Выгрузка запускается из административного контура (Django admin) или через предусмотренный административный интерфейс Telegram-бота.  
  • Требуется корректная обработка пустых значений и безопасная выдача файла только пользователям с нужной ролью.

**7\. Архитектура взаимодействия**

• Telegram-бот передает пользовательские действия на Django backend.  
• Mini App обращается к тому же backend и использует единые данные.  
• Django admin используется для административного просмотра и управления.  
• Все данные хранятся в общей базе данных.

**Принцип реализации:** одна серверная логика, два интерфейса — бот и Mini App.
