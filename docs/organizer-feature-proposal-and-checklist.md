## 1. Проблема «один человек — участник и организатор»

Сейчас у `User` одно поле `role` (`PARTICIPANT` | `CAPTAIN` | `ORGANIZER` | `ADMIN`). Оно описывает **состояние в командной логике** (участник / капитан), а не «должность в оргкомитете».

**Предложение:** не смешивать в одном enum две оси.

| Ось | Смысл | Где хранить |
|-----|--------|-------------|
| Командная роль | участник / капитан | текущее `User.role` (`PARTICIPANT` / `CAPTAIN`; при необходимости `ADMIN` отдельно) |
| Права организатора | может создавать хакатоны и выгружать данные | связь **пользователь ↔ хакатон** (M2M «организаторы этого события») или глобальный флаг «доверенный организатор» + назначение на события |

Итог: пользователь с `role=CAPTAIN` и одновременно в M2M `Hackathon.organizers` — **нормальный кейс**: в Mini App он видит и сценарии капитана, и блок организатора для своих событий.

**Роль `ORGANIZER` в enum:** варианты на выбор (зафиксировать при согласовании):

- **A (рекомендуется):** оставить enum только для командной логики (`PARTICIPANT` / `CAPTAIN`), а `ADMIN` — для Django `is_staff` / суперпользователей; права организатора **только** через M2M/модель хакатона.
- **B:** `ORGANIZER` = «учётная запись без команды, только оргфункции»; капитан-организатор остаётся `CAPTAIN` + M2M (организатор без команды — `ORGANIZER` + M2M).

**Переключение ролей** в продукте — не отдельный «логин», а **контекст UI**:

- Вкладки Mini App: «Профиль», «Поиск команды», при наличии прав — «Мои хакатоны (орг.)» / «Подключение команды к событию».
- В боте: пункт меню «Организатор» только если у `telegram_id` есть хотя бы один хакатон в M2M или глобальное право.

## 2. Модель данных (MVP)

### 2.1. `Hackathon`

- `name`, `slug` (уникальный, для URL/API)
- `description` (опционально)
- `schedule_sheet_url` — ссылка на Google Таблицу с расписанием (как в ТЗ: организатор указывает URL)
- `is_registration_open` — можно ли капитанам подключать команды
- `created_at`, `updated_at`
- `created_by` → `User` (кто создал; не обязательно остаётся в M2M)
- `organizers` — M2M к `User` (кто может редактировать событие и выгружать Excel)

Опционально позже: `starts_at` / `ends_at` для фильтрации и архива.

### 2.2. Связь команда ↔ хакатон

- Модель `TeamHackathon` (или `HackathonTeam`):
  - `team` FK, `hackathon` FK
  - `joined_at`, `status` (`pending` / `active` / `withdrawn`) — на MVP достаточно `active`
  - `UniqueConstraint(team, hackathon)` — одна команда не дублирует подключение к одному событию

Правило: подключать к хакатону может **только капитан** команды (проверка на backend).

### 2.3. Подписка на уведомления по расписанию

- Модель `ScheduleNotificationSubscription`:
  - `user` FK, `hackathon` FK
  - `is_active` (bool)
  - `last_notified_at` / `last_processed_revision` (строка или хеш содержимого) — чтобы не слать дубликаты
  - опционально `timezone` (по умолчанию из настроек или фиксированная, напр. Europe/Moscow)

**Источник событий расписания:** Google Таблица без собственного сервера возможна так:

1. **Публичная таблица** (или доступ по ссылке для чтения): периодически тянуть **CSV export** по стабильному URL вида  
   `https://docs.google.com/spreadsheets/d/{SPREADSHEET_ID}/export?format=csv&gid={GID}`  
   (организатор копирует ссылку на таблицу; в админке или при сохранении парсим `id` и при необходимости `gid` из URL).
2. Парсер CSV: договориться о **формате колонок** (например: `datetime`, `title`, `description`, `notify_minutes_before` или просто `datetime` + `title` и уведомление «за 15 минут» глобально в настройках хакатона).

**Локально без «сервера»:** фоновые задачи запускать через:

- **management command** `sync_schedule_notifications` + вызов вручную при разработке; или
- **django-apscheduler** / **celery** с локальным broker — опционально; для хакатона MVP часто хватает команды + кнопки «проверить расписание сейчас» в боте для отладки.

Отправка уведомлений — через уже существующий Telegram-бот (`send_message` по `telegram_id`).

## 3. API и админка

### 3.1. REST (организатор / капитан / Mini App)

Примерный набор (уточнить при согласовании префикс, напр. `/api/hackathons/`):

| Метод | Назначение |
|--------|------------|
| `GET /api/hackathons/` | Список активных хакатонов (для капитана — подключение) |
| `POST /api/hackathons/` | Создание (только организатор по политике: M2M или staff) |
| `PATCH /api/hackathons/<id>/` | Обновление полей, в т.ч. ссылка на таблицу |
| `POST /api/hackathons/<id>/teams/join/` | Капитан подключает свою команду (`captain_telegram_id`, валидация капитана) |
| `GET /api/hackathons/<id>/export/participants.xlsx` | Выгрузка участников по **всем командам**, подключённым к этому хакатону |
| `GET /api/hackathons/<id>/export/teams.xlsx` | Выгрузка команд хакатона |
| `POST /api/hackathons/<id>/schedule/subscribe/` | Вкл. подписку на рассылку по расписанию |
| `POST /api/hackathons/<id>/schedule/unsubscribe/` | Выкл. |

Защита: для организаторских операций — проверка `request.user` не подходит для Mini App без сессии; **сейчас** у вас передаётся `telegram_id` в теле. Нужно **согласовать**: либо валидация `initData` Telegram Web App + маппинг на пользователя, либо для MVP dev-only с явным предупреждением + whitelist `telegram_id` организаторов в `.env`. Для продакшена — только проверенный `initData` или серверный токен.

### 3.2. Django Admin

- Модели `Hackathon`, `TeamHackathon`, `ScheduleNotificationSubscription` зарегистрированы в admin.
- Action «Скачать Excel» для выбранного хакатона (повтор использования логики экспорта).
- Выгрузки как в текущем ТЗ: участники (ФИО, контакты, стек, статус в команде, **ид хакатона/название**) и команды (название, капитан, состав, вакансии, **привязка к хакатону**).

## 4. Mini App (изменения UX)

1. **Капитан:** экран или секция «Хакатоны» — список событий с открытой регистрацией команд → «Подключить мою команду» (одна активная команда у капитана уже есть в логике проекта).
2. **Участник:** секция «Расписание» — ссылка на таблицу (из выбранного/подключённого хакатона команды пользователя) + переключатель «Уведомлять о событиях из расписания».
3. **Организатор:** экран создания/редактирования хакатона, поле URL таблицы, кнопки выгрузки Excel (или редирект на API с `telegram_id` в query только для dev — нежелательно в проде).

Уточнение: если пользователь в нескольких командах/хакатонах — на MVP выбрать «текущий хакатон» по последнему подключению команды или показать список.

## 5. Бот (Aiogram)

- Команда или кнопка «Мои хакатоны (орг.)» → список + «Выгрузить участников» (отправка файла через `sendDocument`).
- Для подписчиков: фоновая рассылка напоминаний по расписанию (тот же CSV-парсер, что и на backend).
- Локальная отладка: long polling бота + Django на `127.0.0.1`; для Mini App в Telegram нужен **https** (ngrok / cloudflared) — в чеклисте отдельный пункт.

---

## 6. Локальная проверка без своего VPS

- SQLite + `runserver`, бот в polling.
- `.env`: токен бота, `BACKEND_URL` для бота, при необходимости `DJANGO_ALLOWED_HOSTS` + туннель для Mini App.
- Docker Compose (уже в проекте) — опционально поднять всё одной командой.
- Google Таблица: тестовая публичная таблица с 2–3 строками расписания.
- Документировать в README шаг: «как открыть Mini App с туннелем».
